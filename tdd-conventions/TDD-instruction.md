# TDD и unit-тесты для новых Python-сервисов

Принципы и цикл разработки. **Запуск, структура `tests/`, Poetry, пример AAA** — в шаблоне [`tests-README.md`](tests-README.md) (копия лежит в `tests/README.md` каждого сервиса). Здесь — только то, чего нет в том файле.

---

## 1. Стек

### По умолчанию

| Задача | Выбор |
|--------|--------|
| Тесты | **pytest** (функции `test_*`, не классы) |
| Моки | **unittest.mock** |
| Async | **pytest-asyncio** |
| Зависимости | **Poetry**, pytest в `[tool.poetry.group.dev.dependencies]` |
| Подготовка данных | в теле теста или `_Fake*` / `_make_*()` в том же файле |

**Структура каталогов** — зеркало `src/my_package/`, один `test_<module>.py` на модуль; см. `tests-README.md`.

### Опционально — когда уместно

| Инструмент | Кейс |
|------------|------|
| **`@pytest.fixture` / `conftest.py`** | Одинаковая тяжёлая настройка во **многих** тестах домена или сервиса (БД-сессия, клиент, путь к фикстурам на диске). Держите фикстуры локальными: `tests/<домен>/conftest.py`, не глобальные «на всё». |
| **`unittest.TestCase`** | Наследование от существующего базового класса в legacy-модуле; миграция на pytest постепенно. В **новом** коде предпочтительны функции. |
| **`setUp` / `tearDown`** | Редко, внутри TestCase при интеграции с чужим базовым классом. В pytest-стиле чаще `yield`-фикстура с teardown. |
| **`@pytest.mark.parametrize`** | Много однотипных вход→выход (парсеры, валидация, таблицы границ) без копирования тела теста. |
| **`pytest.mark.parametrize` + fixture** | Комбинация таблицы кейсов и общей фикстуры — когда и данных много, и Arrange тяжёлый. |
| **`unittest.mock.patch`** | Глобальный синглтон, модульный импорт «на заморозке», legacy без DI. Предпочтение по-прежнему: передать mock в конструктор/аргумент. |
| **Интеграционные тесты** | Реальная БД/брокер в отдельном каталоге или маркер `@pytest.mark.integration`, не смешивать с unit по умолчанию. |

Ориентир: если без фикстуры тест читается за 15 секунд — не выносить; если десяток тестов дублируют 30 строк Arrange — вынести в фикстуру или `_fakes.py`.

## 2. Читаемый тест (кратко)

- **AAA** — как в `tests/README.md` сервиса (пустые строки между фазами).
- Один тест — одна причина падения; имя `test_<поведение>_<условие>`.
- Assert по **наблюдаемому** результату (ТЗ, пример, ручной расчёт), не копия формулы из кода.
- Внешнее — заглушка через DI; `patch` — узкий, если иначе никак.

**Запуск:** `poetry install` → `poetry run pytest`. Prod без pytest: `poetry install --only main` — в `tests-README.md`.

## 3. Пайплайн TDD (Red → Green → Refactor)

### 0. Узкий срез

Одно предложение без «и»: «пустой список → 0», не «весь экспорт».

### 1. Red — тест первым

Файл по правилам структуры, один тест, AAA. Хороший провал: `ImportError`, `AttributeError`, diff assert. Плохой: опечатка, лишняя фикстура — чинить тест, не код.

`poetry run pytest path::test_name -v` — падение из‑за **отсутствия** функционала.

### 2. Green — минимум

Достаточно пройти assert, вплоть до `return "HELLO"`. Проверяете связку тест ↔ функция.

### 3. Усилить

Второй тест с **другими** входами ломает хардкод → обобщить реализацию → Green.

### 4. Refactor

Без смены поведения; после каждого шага — `pytest`.

### 5. Края

Отдельные тесты: пустой ввод, границы, `pytest.raises`. Опционально в CI: `--cov=src`, смотреть непокрытые ветки в diff, не гнаться за 100%.

## 4. Как тесты что‑то гарантируют

Тесты — **регрессия**, не доказательство. Снижение риска ошибки в тесте:

| Приём | Зачем |
|--------|--------|
| Сначала Red | Зелёный с нуля = тест ни о чём |
| Два разных входа | Ломают `return 42` |
| Ожидание из ТЗ/эталона | Не дублировать формулу из кода |
| Один Act на тест | Понятно, что сломалось |
| Сомнение | `print` / `-s` / `--pdb` после Act; испортить expected → снова Red |

| Симптом | Скорее | Действие |
|---------|--------|----------|
| `ImportError` до assert | Нет кода/пути | Модуль, `pythonpath` |
| Стабильно «не тот» результат | Неверный expected | ТЗ, правка теста |
| Падение после рефакторинга | Сломали код | Чинить код |
| Flaky | Время/сеть/порядок | Мок I/O, фиксированные данные |

Unit-тесты — чистая логика и явные заглушки; БД, брокеры, чужие HTTP — реже, отдельно.
